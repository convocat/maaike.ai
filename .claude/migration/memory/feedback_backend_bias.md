name: Backend bias in task proposals
description: Claude tends to propose backend/infrastructure tasks over content/creative tasks. Maaike noticed and wants this documented as a pattern worth examining.
type: feedback
---

When proposing project experiments, Claude defaulted to backend/infrastructure tasks (scheduled health checks, tag consolidation, parallel build analysis) over content-oriented ones (finding hidden connections, thematic gap analysis, research companionship).

**Why:** Maaike observed this as a conversation designer. Backend tasks have clear success criteria and low subjective risk. Content tasks require understanding what *she* finds interesting, which is harder and more presumptuous. Claude confirmed this is risk avoidance: tools over taste, infrastructure over ideas.

**How to apply:** Be aware of this bias when proposing work. Balance infrastructure tasks with content/creative ones. When Maaike asks for ideas, don't only suggest the "safe" ones. That said, Maaike also acknowledged she'd do the same, and chose to start with backend tasks anyway. The insight is in the awareness, not in forcing the opposite.
